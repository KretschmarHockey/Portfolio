Enemy Movement

September 23

I wanted the standard enemy ships to look like a human could make the moves. I wanted sudden change in directions sometimes. The algorithm design I went with separates the X and Y coordinates when it is being processed.

A random X and Y coordinate is generated as well as a random X and Y velocity. The enemy ship then travels to the coordinates at the velocity. Due to the fact that the coordinates and velocity is random, the enemy ship can arrive at the X coordinate before it arrives to the Y coordinate. If this happens, the enemy ship stays at that X coordinate, until the Y coordinate is reached. Once the X and Y coordinates are reached, then a random X and Y coordinate and X and Y velocity is generated again.

High Score

September 27

I created a Scoreboard that can be displayed. The scores are stored in an .ini file. At the moment the scores are not updated. I used a lot of time (3+ hours) to get this working. First I struggled to find a way to store the scoreboard to be read from the Scoreboard object. My knowledge of passing by reference really affected the time. I kept passing on pointers to store in vectors to be read. However, when I would change to the next line of the .ini file it would change all the values in the vectors. This has made me learn about being careful of passing by reference and pointer. The next issue I had was that the way I was storing the highscores was creating memory leaks. I ended up having to completely redo how the highscores were stored. The game object now stores the highscores and can be accessed by the Scoreboard object.
